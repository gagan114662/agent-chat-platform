# Design Spec — Plan 2.1a: Channels & Threads Navigation

**Status:** 🟢 Scoped (author's call — see "Scope decisions"); ready for planning
**Date:** 2026-06-03
**Parent design:** [`2026-06-03-agent-chat-fusion-design.md`](./2026-06-03-agent-chat-fusion-design.md)
**Builds on:** Plan 2.0a (backend) + 2.0b (UI), merged to `main`.

---

## 1. Goal

Plan 2.0 shipped a single hard-coded demo thread. **2.1a turns it into a real multi-thread
workspace:** list channels, list/open threads within a channel, and create a new thread (optionally
bound to a repo so `@mention`s can run the fusion loop there). This is the first slice of the broader
Phase 2.1 ("breadth"); DMs, the task board, and search follow in 2.1b+.

## 2. Scope decisions (taken, not asked)

**In scope (2.1a):**
- **Backend** (`services/app`): read channels, read/create threads in a channel, read repos (for the
  new-thread repo picker). Org-scoped via the existing dev-auth header stub (`x-org-id`).
- **UI** (`services/web`): dynamic sidebar (channels → threads from the API), switching the active
  thread, and a "New thread" form (title + optional repo). The active thread drives the existing
  `useThreadStream`/`ThreadView`.

**Out of scope (deferred):**
- Creating/renaming/archiving **channels** (the seed's `general` channel is enough for 2.1a) → 2.1b.
- **DMs**, **task board**, **message search** → 2.1b/2.1c.
- Real auth/RBAC (still the dev stub) → 2.2.
- Pagination/infinite scroll of messages → later (threads are small in dev).

## 3. API (additions to `services/app`)

All org-scoped by the `x-org-id` dev header (default `o1`). JSON.

| Method · Path | Returns | Notes |
|---|---|---|
| `GET /channels` | `Channel[]` | channels in the actor's org |
| `GET /channels/:id/threads` | `Thread[]` | threads in a channel (newest-first) |
| `POST /channels/:id/threads` `{title, repoId?}` | `Thread` (201) | creates a thread; `repoId` optional, validated to the same org if present |
| `GET /repos` | `Repo[]` | repos in the actor's org (for the new-thread picker) |

`Channel`, `Thread`, `Repo` shapes are the existing Drizzle rows (no schema changes — the tables
already exist from 2.0a). Thread create generates an id app-side (`randomUUID`).

## 4. Components (additions to `services/web`)

- **`api.ts`** gains `listChannels()`, `listThreads(channelId)`, `createThread(channelId, {title, repoId?})`, `listRepos()`.
- **`Sidebar`** becomes data-driven: fetches channels + threads, renders the tree, highlights the
  active thread, calls `onSelectThread(id)`. Hosts a **`NewThreadForm`** (title input + repo `<select>`
  + create button) that POSTs and selects the new thread.
- **`App`** holds `activeThreadId` state (defaults to the first thread, or the seeded `t1`), passes it
  to `useThreadStream` and `ThreadView`, and to `Sidebar` for highlighting + selection.

No change to `useThreadStream` (already keyed by `threadId`), `MessageItem`, `Composer`, or the
backend fusion path.

## 5. Testing

- **Backend:** module functions (`listChannels`, `listThreads`, `createThread`, `listRepos`) against
  the test Postgres; route tests for `GET /channels`, `GET /channels/:id/threads`, `POST .../threads`
  (incl. repo-org validation), `GET /repos`.
- **UI:** `Sidebar` renders fetched channels/threads and calls `onSelectThread` on click; `NewThreadForm`
  posts and surfaces the created thread; `App` smoke test still renders with the dynamic sidebar
  (fetch stubbed).

## 6. Definition of Done (2.1a)

Backend unit/route tests green (Postgres); web component tests green + `pnpm build` clean. With the
stack running, the sidebar lists channels and threads, you can open any thread, and "New thread"
creates one (optionally repo-bound) and switches to it — then `@mention`s in that thread run the
existing fusion loop. Channel management, DMs, task board, and search remain deferred.
