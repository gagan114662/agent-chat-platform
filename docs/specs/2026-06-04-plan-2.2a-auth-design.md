# Design Spec — Plan 2.2a: Session-Based Authentication

**Status:** 🟢 Scoped (author's call); ready for planning
**Date:** 2026-06-04
**Parent:** [`2026-06-03-agent-chat-fusion-design.md`](./2026-06-03-agent-chat-fusion-design.md)
**Builds on:** Phase 2.1 (merged to `main`).

## 1. Goal
Replace the `x-org-id`/`x-user-id` dev-header stub with **real server-side sessions**: log in as a
member, get an opaque session token, and resolve the request principal from it. This is the auth
*backbone*; **RBAC/permissions** are 2.2b and **passwords/SSO** are 2.2c.

## 2. Scope decisions (taken)
**In:** a `sessions` table; opaque session tokens (server-side, revocable, expiring); `POST /auth/login`
(dev: pick a member by id), `POST /auth/logout`, `GET /auth/me`, `GET /auth/members` (dev login picker);
a Fastify `preHandler` that resolves `Authorization: Bearer <token>` → `req.principal`; a UI login gate.
**Migration safety:** the existing `actor(req)` keeps a **dev-header fallback** when no valid session is
present, so existing route tests/dev keep working. The fallback is explicit and slated for lock-down in
2.2c (behind a prod flag).
**Out (→ later):** passwords/SSO/OAuth (2.2c), RBAC/permission checks (2.2b), per-agent auth, session
refresh/rotation, CSRF (bearer-token, not cookie, so N/A for now).

## 3. Schema (migration)
`sessions(id text PK, member_id text NOT NULL, org_id text NOT NULL, created_at timestamptz DEFAULT now(),
expires_at timestamptz NOT NULL)`. Token = the opaque `id` (a UUID). Add `sessions` to the test
harness truncate list.

## 4. API (additions to `services/app`)
| Method · Path | Returns | Notes |
|---|---|---|
| `GET /auth/members` | `{id,displayName,orgId}[]` | dev login picker (lists members) |
| `POST /auth/login` `{memberId}` | `{token, member}` (201) | creates a 7-day session; 400 if member unknown |
| `GET /auth/me` | `{orgId,userId}` or **401** | the resolved principal |
| `POST /auth/logout` | 204 | deletes the bearer's session |

A root `preHandler` resolves `Bearer <token>` → `req.principal = {orgId,userId}` (when valid). `actor(req)`
returns `req.principal` if set, else the dev-header fallback.

## 5. UI (additions to `services/web`)
- `authHeaders()` sends `Authorization: Bearer <token>` (from `localStorage`) when present, else falls
  back to the dev headers. All API calls use it.
- `useAuth` hook: on load calls `GET /auth/me`; exposes `{principal, loading, login(memberId), logout()}`.
- `LoginScreen`: lists `/auth/members` as buttons → `login` stores the token and enters the app.
- `App` gates on `useAuth`: loading → spinner; no principal → `LoginScreen`; authed → the workspace
  (with a sign-out affordance showing the current user).

## 6. Testing
- **Backend:** `auth` module (`createSession`/`resolveSession` valid+expired+unknown, `deleteSession`,
  `listMembersForLogin`); auth routes via `app.inject` (login→me 200; no token→me 401; logout→me 401).
  Existing route tests (header path) must stay green.
- **UI:** auth api (login stores token; `me`); `LoginScreen` (renders members, click logs in);
  `App` gate (shows login when `me` 401, shows workspace when authed).

## 7. Definition of Done (2.2a)
App backend suite green (Postgres + migration; existing header-path tests intact) + web suite green +
build clean. Live: the UI shows a login screen, signing in as a seeded member enters the workspace with
all requests bearer-authenticated, and sign-out returns to login. RBAC and passwords/SSO remain for 2.2b/2.2c.
