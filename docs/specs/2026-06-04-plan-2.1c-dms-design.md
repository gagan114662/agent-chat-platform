# Design Spec — Plan 2.1c: Direct Messages

**Status:** 🟢 Scoped (author's call); ready for planning
**Date:** 2026-06-04
**Parent:** [`2026-06-03-agent-chat-fusion-design.md`](./2026-06-03-agent-chat-fusion-design.md)
**Builds on:** Plan 2.1b, merged to `main`.

## 1. Goal
Let a user open a **direct message** with another principal — a human member or an **agent** — reusing
the existing thread/message/WebSocket machinery. This completes the Phase-2.1 "breadth" set.

## 2. Scope decisions (taken)
**In:** list principals (members + agents) in the org; get-or-create a DM thread with a chosen
principal; list DM threads; a "Direct Messages" section + new-DM picker in the sidebar.
**Out (→ later):** group DMs, multi-user pairing semantics (one dev user for now), repo-bound fusion
inside a DM (DMs are pure chat for 2.1c — channel threads remain the fusion surface), unread/badges,
real auth (still the dev stub).

## 3. Schema change (migration)
Extend `threads` to represent DMs as channel-less threads:
- `channelId` → **nullable** (DMs have none).
- add `kind text NOT NULL DEFAULT 'channel'` (`'channel' | 'dm'`).
- add `dmPeerKind text` (`'human' | 'agent'`, dm only) and `dmPeerId text` (nullable).

Channel listing (`GET /channels/:id/threads`) is unaffected (filters by `channelId`, so DMs are excluded);
the mention→fusion route already no-ops when `repoId` is null, so DM messages are pure chat.

## 4. API (additions to `services/app`)
Org-scoped via the `x-org-id` dev header; actor = `x-user-id` (default `m1`).

| Method · Path | Returns | Notes |
|---|---|---|
| `GET /principals` | `Principal[]` | members + agents in the org, excluding the actor (you can't DM yourself) |
| `GET /dms` | `Thread[]` | the org's DM threads, newest-first |
| `POST /dms` `{peerKind, peerId}` | `Thread` (201) | get-or-create the DM with that principal; 400 if the principal doesn't exist |

`Principal = { kind: 'human'|'agent', id, name }`. DM `Thread` rows carry `kind:'dm'`, `dmPeerKind`,
`dmPeerId`, `title = peer's display name`, `channelId = null`.

## 5. UI (additions to `services/web`)
- **`Sidebar`** gains a **Direct Messages** section listing DM threads (selectable like channel threads),
  plus a **`NewDmPicker`** (a `<select>` of principals → `onStartDm(peerKind, peerId)`).
- **`App`** fetches `principals` + `dms`, renders them, and `onStartDm` posts then selects the DM thread.
  DM threads use the same `useThreadStream`/`ThreadView`/`Composer` as channel threads.

## 6. Testing
- **Backend:** `listPrincipals` (members+agents, excludes self), `getOrCreateDm` (creates with peer name;
  idempotent for the same peer; 400 on unknown), `listDms` module tests; routes via `app.inject`.
- **UI:** `NewDmPicker` (selecting a principal calls `onStartDm`); `Sidebar` renders the DM section;
  `App` smoke still renders with the added fetches stubbed.

## 7. Definition of Done (2.1c)
App backend suite green (Postgres + migration) + web suite green + build clean. Live: a "Direct Messages"
section lists DMs; picking a principal opens (or reuses) a DM thread you can chat in via the same
real-time stream. Group DMs, fusion-in-DM, and real auth remain future work.
