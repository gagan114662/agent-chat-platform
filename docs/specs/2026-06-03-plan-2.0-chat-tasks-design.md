# Design Spec — Plan 2.0: Chat + Tasks Core (the fusion seam, through chat)

**Status:** 🟢 Approved-in-brainstorm — ready for implementation planning
**Date:** 2026-06-03
**Parent design:** [`2026-06-03-agent-chat-fusion-design.md`](./2026-06-03-agent-chat-fusion-design.md)
**Builds on:** Plan 1 (Fusion Engine walking skeleton — `merged` to `main`, live-proven)

---

## 1. Goal & wedge realized

Plan 1 proved `intent → sandbox → agent → PR → auto-merge` as a callable workflow. **Plan 2.0
makes that loop reachable from a chat thread**, with agents as first-class principals:

> A human posts `@coder fix the login bug` in a thread → the backend persists the message,
> creates a **Task** and a **Run**, and starts the **Plan-1 `fusionWorkflow`** → step-by-step
> progress streams back into the thread live → a final **PR card** (number, URL, checks, outcome)
> is posted and the Task moves to `DONE`/`BLOCKED`. The agent authors those messages and owns the
> Task — it is a principal, not a bot-user.

This is the first **demoable** cut of the product: chat → execution → PR → back to chat, on one
surface. It is the walking skeleton for the "full chat+tasks experience"; breadth and hardening
come in Phases 2.1–2.3.

## 2. Scope

**In scope (2.0):**
- One **org**, one **workspace**, **channels → threads → messages**, persisted in Postgres.
- **Agents as principals**: an `Agent` row with identity; messages and Tasks can be authored/owned
  by either a `Human` or an `Agent`.
- **`@mention` parsing** in a message → resolves to an Agent permitted on the thread's repo.
- **Task + Run lifecycle** linked to the thread (per parent spec's data model).
- **Mention → fusion wiring**: a **Temporal worker** (new — Plan 1 never built one) that registers
  `fusionWorkflow` + activities; the backend starts a workflow per Run.
- **Step-event progress streaming**: the fusion flow emits lifecycle events (sandbox started, PR
  opened, each checks-poll result, merged/failed/timeout) → relayed into the thread over WebSocket.
- **React web UI**: workspace → channel list → thread view; post a message; see messages + live
  agent progress + the final PR card.
- **Stubbed dev auth** (pick a user/org for the session).
- Runs locally via **docker-compose** (Postgres, Temporal, Plan-1 Go sandbox-runner, the app).

**Out of scope (deferred — phase noted):**
- DMs, channel/workspace admin UI, task-board view, message search/history pagination → **2.1**.
- Real SSO, human + agent RBAC, per-repo agent permission management UI → **2.2**.
- NATS event backbone, presence, reconnection/replay, multi-org Postgres RLS *enforcement*,
  K8s deploy → **2.3**.
- Real agent adapters (Claude Code/Codex/…) — **keep Plan 1's Go `FakeAgent`**. → later plan.
- Fine-grained agent **stdout** streaming (needs the Go sandbox-runner to stream; 2.0 streams
  coarse workflow **step** events only). → later plan.
- CI-resolution fix-on-red loop, risk router, QA-for-UI gate → Plan 4 (per parent spec).

## 3. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Backend shape | **Single modular TS app** (`services/app`), modules: `chat`, `tasks`, `agents`, `realtime`, `fusion-bridge` | Avoid premature microservice/NATS complexity; split out in 2.3. |
| API surface | **REST** (CRUD + actions) + **WebSocket** (live thread events) | Simple, well-understood; WS carries the realtime stream. |
| Realtime transport | **WebSocket + Postgres `LISTEN/NOTIFY`** fan-out | YAGNI vs. NATS for one process; NATS is a 2.3 item. |
| Persistence | **Postgres + Drizzle ORM** (SQL-first, typed); migrations checked in | Lightweight, plays well with raw `LISTEN/NOTIFY`; `org_id` on every row from day one. |
| Tenancy | `org_id` column on every row; **RLS deferred** to 2.3 | Columns now (cheap, correct shape); enforcement when multi-tenant. |
| Auth | **Stubbed dev login** (header/cookie selects `{userId, orgId}`) | Don't block the seam; real SSO is 2.2. |
| Execution | **Plan-1 `fusionWorkflow`** via a new **Temporal worker** | Reuse proven loop; add the missing worker entrypoint. |
| Progress | **Step events** emitted by the fusion flow → `run_event` table + `NOTIFY` → WS | Honest given sandbox-runner is synchronous today. |
| Agent | **Plan-1 Go `FakeAgent`** | Proving the seam, not agent quality. |
| Frontend | **React + Vite + TS + Tailwind**, WS client | Default, fits the TS monorepo, fastest to a chat UI. |
| Run target | **docker-compose** | Local & reproducible; K8s later. |

## 4. Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│  React UI (Vite)  — channel list · thread view · composer · PR card│
└───────────────▲───────────────────────────────┬──────────────────┘
        REST (fetch)                      WebSocket (live thread events)
                │                                 │
┌───────────────┴─────────────────────────────────▼──────────────────┐
│  services/app  (single TS process)                                   │
│   chat        channels/threads/messages, @mention parse              │
│   tasks       Task + Run lifecycle state machine                     │
│   agents      Agent principals, repo permission check                │
│   realtime    WS server; Postgres LISTEN → fan-out to subscribers    │
│   fusion-bridge  on mention: create Task+Run, start fusionWorkflow;  │
│                  subscribe to run_event → post thread messages        │
└──────┬─────────────────────────────▲───────────────────────┬────────┘
       │ Drizzle                       │ run_event NOTIFY       │ Temporal client
       ▼                               │                        ▼
   ┌────────┐                          │            ┌───────────────────────┐
   │Postgres│◄─────────────────────────┘            │ Temporal (dev server) │
   └────────┘    activities write run_event         │  worker: fusionWorkflow│
                                                     │   + activities         │
                                                     └──────────┬─────────────┘
                                                                │ HTTP POST /run
                                                                ▼
                                                   ┌──────────────────────────┐
                                                   │ Go sandbox-runner (Plan 1)│
                                                   │  clone · FakeAgent · push │
                                                   └──────────────────────────┘
```

For 2.0 the **Temporal worker runs in-process** with `services/app` (one container, started
alongside the HTTP/WS server) — simplest to run and debug. Splitting the worker into its own
process/deployment is a 2.3 scaling item. The activities reach Postgres to write `run_event`s and
reach the Go sandbox-runner + GitHub exactly as in Plan 1.

## 5. Data model (2.0 subset)

Per the parent spec, narrowed to what 2.0 needs. Every table carries `org_id`.

```
Org(id, name)
 └─ Workspace(id, org_id, name)
     ├─ Channel(id, org_id, workspace_id, name)
     │   └─ Thread(id, org_id, channel_id, title, repo_id)
     │       └─ Message(id, org_id, thread_id, author_kind: 'human'|'agent',
     │                  author_id, body, created_at, kind: 'chat'|'system'|'pr_card',
     │                  metadata jsonb)
     ├─ Repo(id, org_id, workspace_id, github_owner, github_name,
     │        default_branch, token_ref)         -- token_ref names an env var holding the
     │                                            -- GitHub token in 2.0 (Vault/secrets = later);
     │                                            -- merge policy columns land in 2.2
     ├─ Member(id, org_id, workspace_id, display_name)        -- Human principal
     └─ Agent(id, org_id, workspace_id, handle, display_name,
              adapter: 'fake', config jsonb)                  -- Agent principal

Task(id, org_id, thread_id, title, state: 'open'|'in_progress'|'done'|'blocked',
     assignee_kind, assignee_id, created_by_kind, created_by_id)
 └─ Run(id, org_id, task_id, state: 'pending'|'running'|'merged'|'checks_failed'
        |'timeout'|'error', workflow_id, branch, commit_sha, pr_number, pr_url)
     └─ RunEvent(id, org_id, run_id, seq, type, payload jsonb, created_at)
```

- **`author_kind`/`assignee_kind` + id** = the polymorphic principal (Human|Agent), making agents
  first-class without a second-class "bot user".
- **`Run` is the unit of execution & audit** (parent spec): one Task → many Runs; each Run owns one
  workflow and at most one PR. `RunEvent` is the append-only step log that also drives the live UI.
- `Message.kind = 'pr_card'` with structured `metadata` renders the final PR result card.

## 6. The mention → fusion flow (2.0)

```
[1] POST /threads/:id/messages { body: "@coder fix the login bug" }
[2] chat: persist Message(author=human). realtime: NOTIFY → WS broadcast to thread.
[3] chat: parse @mentions → agents. For each permitted agent on thread.repo:
[4] tasks: create Task(in_progress) + Run(pending), workflow_id = `run-<runId>`.
[5] fusion-bridge: start Temporal fusionWorkflow with
      { owner, repo, repoUrl, baseBranch, intent: <message body>, branch: agent/<runId> ,
        githubToken (from Repo.token_ref), sandboxUrl, pollMs, maxPolls }.
[6] worker runs the Plan-1 loop. At each step the activity appends a RunEvent
      (sandbox_started, branch_pushed, pr_opened, checks: <state>, outcome: <…>)
      and NOTIFYs. fusion-bridge (LISTEN) turns each into a thread Message
      (kind 'system', authored by the Agent) → WS → UI shows live progress.
[7] terminal: Run + Task state updated; a final Message(kind 'pr_card') is posted;
      Task → done (merged) or blocked (checks_failed/timeout/error).
```

**Plan-1 change required (small, backward-compatible):** `runFusion` gains an optional
`onEvent(event)` hook (in `FusionDeps` or `FusionOptions`) called at each step. Existing Plan-1
unit tests pass an undefined/no-op emitter and are unaffected; the 2.0 activity passes an emitter
that writes `RunEvent`s. No change to the Go sandbox-runner in 2.0.

## 7. Component responsibilities (interfaces, not internals)

- **`chat`** — `createMessage`, `listMessages`, `parseMentions(body) → handles[]`; owns
  channels/threads/messages. Depends on: Drizzle, `realtime` (to notify).
- **`agents`** — `resolveMention(handle, orgId) → Agent`, `isPermittedOnRepo(agentId, repoId)`.
- **`tasks`** — `openTaskForMention(...) → {task, run}`, `transitionRun(runId, state, fields)`,
  the Run state machine. Single source of truth for Task/Run state.
- **`fusion-bridge`** — `startRun(run)` (Temporal client), `onRunEvent(evt)` (LISTEN handler →
  post thread message + transition Run). The only module that knows about Temporal.
- **`realtime`** — WS endpoint `/ws?threadId=…`; Postgres `LISTEN` on a channel; fan-out to
  subscribers. Pure transport — no business logic.
- **`temporal/worker`** — registers `fusionWorkflow` + activities; the activity injects the
  `RunEvent`-writing emitter into `runFusion`.
- **UI** — `ChannelList`, `ThreadView` (message list + live event items + PR card), `Composer`.
  Talks REST for history, WS for live; optimistic append on send.

Each module is independently testable with fakes (the pattern Plan 1 validated).

## 8. Error handling

- Per parent spec — **fail loudly in the thread**. Every failure becomes a `system` Message + a
  Run/Task state: provision/clone fail → `error` + posted reason; checks fail → `checks_failed` +
  `blocked`; budget/timeout → `timeout` + `blocked`.
- **Temporal durability**: worker/app restart mid-Run resumes the workflow; `RunEvent.seq` +
  idempotent "post message for event" keep the thread consistent (no dupes on replay).
- **Idempotency**: external actions remain keyed by Run id (inherited from Plan 1).
- **Secret hygiene** (carry-over from Plan 1 review): sanitize token-bearing repo URLs out of any
  error surfaced into a thread message. (The sandbox-runner sanitization itself is a Plan 3 item;
  2.0 must at minimum not echo a raw token into chat.)

## 9. Testing strategy

- **Unit:** mention parser; Run state machine; `chat`/`tasks`/`agents` modules with a fake DB or a
  test Postgres; the `onEvent` emitter mapping → `RunEvent`s.
- **Integration:** spin up Postgres (testcontainers or compose) — post a message with a mention,
  assert Task+Run created, workflow started (Temporal time-skipping env + stubbed activity, as in
  Plan 1's Task 6), `RunEvent`s written, terminal `pr_card` message posted, Task `done`.
- **Realtime:** a WS client receives the live event sequence for a thread in order.
- **E2E (env-gated, opt-in):** reuse the Plan-1 fixture repo + sandbox-runner — post a real mention
  through the running app and assert a real PR is opened/merged and the thread shows the loop.
  Skipped without env (same gate style as Plan 1's Task 7).
- **UI:** component tests for `ThreadView` rendering of message/event/PR-card types.

## 10. Definition of Done (2.0)

`docker-compose up` brings up Postgres + Temporal + sandbox-runner + app + UI. In the browser, a
user posts `@coder <intent>` in a thread against a configured repo, watches live step progress,
and sees a final PR card; with the always-green fixture the PR auto-merges and the Task shows
`DONE` — the full chat→execution→PR→chat seam, proven end-to-end. Unit + integration suites green;
the env-gated chat-e2e produces a real merged PR.
